# Replays: the seed-replay contract

Because the engine is fully deterministic (see [engine.md](engine.md):
[`bt_core::Game`](../bt-core/src/game.rs) is seeded with a `drand48` LCG and advanced
by an explicit fixed-`dt_ms` clock, and Ernie ([`bt-ai`](../bt-ai)) is a pure
function of the seed), a complete recording of a game is **just the seed plus the
timestamped sequence of inputs**. Everything else — gravity, the AI's moves, the RNG,
the cross-player relay — is regenerated by re-running the engine.

This is a **seed replay**, not a state recording. That single object plays a triple
role and is the foundation for online recordability. The crate is
[`bt-replay`](../bt-replay/src/lib.rs).

---

## The contract

A practice / vs-computer recording is a [`Replay`](../bt-replay/src/lib.rs):

```text
{ version, seed, mode, ai_level, dt_ms, engine_sha, tick_count, frames: [(tick, Input)], title }
```

- **`seed`** — the `drand48` seed; fully determines the piece stream and all
  randomized effects.
- **`mode`** — [`Mode`](../bt-replay/src/lib.rs): `Practice`, `VsComputer`, or
  `VsPlayer`.
- **`ai_level`** — Ernie's difficulty (vs-computer only; regenerated from the seed).
- **`dt_ms`** — the fixed timestep the engine was advanced with. Faithful replay
  *requires* a fixed timestep (an accumulator loop, not wall-clock deltas);
  [`ReplayPlayer`](../bt-replay/src/lib.rs) advances in exactly these steps.
- **`engine_sha`** — the engine build this was recorded against (`git` short SHA, or
  `"dev"` locally). See the mismatch discussion below.
- **`frames`** — each [`Frame`](../bt-replay/src/lib.rs) is an [`Input`] stamped with
  the tick index it was applied at (the number of engine ticks already executed when
  it landed).

Re-running a fresh engine with the same seed and re-applying the inputs at the same
tick boundaries reproduces the game **bit-for-bit**. This is the core determinism
guarantee, pinned by `practice_replay_is_bit_exact` and
`vs_computer_replay_is_bit_exact` in
[`bt-replay/tests/pbt_replay.rs`](../bt-replay/tests/pbt_replay.rs) (and the broader
suite in [testing.md](testing.md)).

### `engine_sha`-mismatch: a feature, not a bug

A seed replay records only inputs and regenerates everything else by re-running the
engine, so it is faithful only on the **same engine build**. That is precisely the
point for debugging: replay the same inputs on a *different* `engine_sha` to see
where behavior diverged. `engine_sha` is recorded so playback can flag a mismatch
("this was recorded on commit X; you're on commit Y") rather than silently producing
a different result. The server stamps it at record time from the compile-time
`BT_GIT_SHA` (the `ENGINE_SHA` constant in
[`bt-server/src/main.rs`](../bt-server/src/main.rs)); `bt-wasm` does the same for
client-side recordings.

## The triple role

One object serves three purposes (from the
[`bt-replay`](../bt-replay/src/lib.rs) `//!` header):

1. **Bug-report trace.** The "report a bug" button captures the current game as a
   replay, so a maintainer can *watch* exactly what the player did. The
   [`dump_replay`](../bt-replay/examples/dump_replay.rs) example (see
   [engine.md](engine.md)) replays one headlessly and scans every tick for engine
   invariant violations.
2. **Library content.** Stored recordings populate the replay library / showcase.
   `title` is an optional human label (e.g. a weapon-showcase name); it defaults to
   `None` on older recordings.
3. **Deterministic test case.** A failing seed becomes a regression: "does this bug
   still reproduce on commit X?" The **fuzz→replay bridge**
   ([`bt-replay/tests/fuzz_replay.rs`](../bt-replay/tests/fuzz_replay.rs)) turns any
   fuzz seed into a faithful, scrubbable replay — see below.

### `ReplayPlayer` and scrubbing

[`ReplayPlayer`](../bt-replay/src/lib.rs) drives a `Replay` over the engine at its
recorded timestep. `step()` applies every input stamped at the current tick then
ticks the engine once; `run_to_end()` runs headless; `seek(target)` jumps to a tick
(backward seeks rebuild from the seed and fast-forward — cheap, since the engine
runs thousands of ticks per millisecond). This is what a scrubber/seek bar calls. A
vs-computer replay reconstructs both the human board and Ernie's, plus the match
result.

## Online (server-authoritative) recordability — closing D5

The original gap **D5** was that online games weren't replayable: with a P2P relay,
reproducing a match meant recording all of its effects. The authoritative
client/server model (see [architecture-netcode.md](architecture-netcode.md)) fixes
this structurally: the server sees a **totally-ordered input log**, so an online
match is recorded as a [`VersusReplay`](../bt-replay/src/lib.rs):

```text
{ version, seed_a, seed_b, dt_ms, engine_sha, tick_count, frames: [(tick, side, Input)], title }
```

Just the two seeds plus the totally-ordered client-input stream. Because
[`Versus`](../bt-core/src/versus.rs) is deterministic, replaying = re-running it and
applying each input at its tick — **the whole relay (weapons, taxes, bazaar, spies)
reproduces exactly, so none of it needs to be recorded**. Only client *inputs* are
in the log; server-only actions (the admin grant, for example) are deliberately
*not* recorded, so a replay reproduces the bout as if those never happened.

[`Bout::to_replay`](../bt-server/src/bout.rs) exports the live match;
[`VersusReplayPlayer`](../bt-replay/src/lib.rs) plays it back over a `Versus`
(stepping/seeking like `ReplayPlayer`, reproducing **both** boards). The
tick-stamping contract is exact: the server loop applies a batch of inputs then
*always* ticks, so every recorded frame is stamped at a tick strictly less than the
final tick count and the player replays all of them — explained in the
`to_replay` doc comment in [bout.rs](../bt-server/src/bout.rs). Round-trip +
deterministic-reproduction is pinned by
`versus_replay_round_trips_and_reproduces_deterministically`.

## The fuzz → replay bridge

`bt-ai`'s weapon fuzzer drives Ernie while injecting random weapons and asserts
invariants — but a failing seed was only reproducible as a *number*, not something
you could *watch*. [`fuzz_replay.rs`](../bt-replay/tests/fuzz_replay.rs) records the
same run (the AI's exact moves + the weapon injections) into the replay format, so
any fuzz seed becomes a faithful, scrubbable replay you can open in the library.

The catch the format had: the AI hard-drops with
[`Game::ai_begin_drop`](../bt-core/src/game.rs) (a flat score bonus), not the human
[`begin_drop`](../bt-core/src/game.rs) (which scores `BT_BOARD_HGT - y`). The
dedicated [`Input::AiDrop`](../bt-replay/src/lib.rs) variant closes that, so an
AI-driven recording replays bit-for-bit. `AiDrop` is one of the relay-internal
inputs a *client* may never send (see
[`is_legal_client_input`](../bt-server/src/bout.rs)) — it only exists for faithful
recordings.

## Storage: SQLite (`REPLAY_DB`) and routes

The server persists recordings in SQLite at the path from the **`REPLAY_DB`** env var
(default `replays.db`; prod points it at the attached fly volume). The schema stores
the replay JSON blob plus indexed columns (mode, seed, ai_level, tick_count, inputs,
engine_sha, created_at, title, lines, and the player names for versus replays). The
routes (in [`bt-server/src/main.rs`](../bt-server/src/main.rs)):

| Route | Purpose |
|---|---|
| `POST /api/replays` | Store a recording (validates it parses as a `Replay` first, so junk is never persisted); returns `{"id": "..."}`. |
| `GET /api/replays/:id` | Fetch a stored recording as JSON (injects the player names from DB columns into the blob so the viewer can label boards). |
| `GET /api/replays` | List recordings (newest first, capped) with browse metadata — backs the library page. |
| `GET /replay/:id` | The pretty, shareable playback link; redirects to the static player page, which fetches `/api/replays/:id`. |

Legacy JSON recordings predate the `title` field, so it `#[serde(default)]`s to
`None` on deserialize. Spectating a live match (the read-only two-board `spectate`
view) and the leaderboard are separate paths; see `docs/deployment.md` for the
admin/ops surface. The version field is `REPLAY_VERSION`, bumped on any
incompatible on-disk format change.
